# 1. 시스템 영향 및 범위

## 배경 및 목표

| 항목 | 명세 |
|---|---|
| 배경 | 기존 `reality-game`은 웹앱 중심으로 구현되어 있으나, 새 목표는 디스코드 기반 게임이다. |
| 목표 | 디스코드 안에서 세션을 시작하고, 미션을 진행하고, 복선을 쌓고, 엔딩을 생성하는 게임을 만든다. |
| 서비스 형태 | 웹 화면이 아니라 디스코드 상호작용을 중심으로 동작한다. |
| 복원 목표 | 재시작 후에도 저장된 세션을 다시 불러올 수 있어야 한다. |

## 범위

| 포함 | 제외 |
|---|---|
| 디스코드 세션 시작과 종료 | 브라우저 전용 게임 화면 |
| 미션 선택, 완료, 체크포인트, 엔딩 | 웹 폼 기반 입력 흐름 |
| 복선 기록과 회수 | 웹 localStorage만 사용하는 저장 |
| 안전 필터와 범용 미션 경로 | 민폐, 불법, 위험 미션 허용 |
| 세션 저장과 복원 | 사용자 계정/로그인 |
| 개발/배포 실행 준비 | 관리자 편집기, 추천 고도화 |

## 시스템 영향

| 영역 | 영향 |
|---|---|
| 진입점 | 웹 루트 화면 대신 디스코드 명령과 인터랙션이 진입점이 된다. |
| 상태 관리 | 브라우저 상태가 아니라 디스코드 세션 상태를 관리해야 한다. |
| 저장 | 세션과 복선, 완료 이력은 재시작 후 복원 가능한 저장소에 저장한다. |
| 콘텐츠 | 미션 데이터와 이야기 회수 규칙은 공용 도메인 로직으로 분리한다. |
| 운영 | 봇 토큰, 환경 변수, 배포 설정이 필요하다. |

## 수정 대상 및 참고 목록

| 대상 | 작업 |
|---|---|
| 제품 진입 구조 | 웹 중심 진입을 제거하고 디스코드 명령 진입으로 바꾼다. |
| 세션 상태 계층 | 세션 생성, 진행, 복원, 리셋을 담당하는 상태 계층을 둔다. |
| 미션/엔딩 도메인 | 미션 선택, 완료 기록, 복선 회수, 엔딩 생성을 분리한다. |
| 저장 계층 | 세션 데이터를 직렬화해 저장하고 복원한다. |
| 환경 설정 | 디스코드 봇 실행에 필요한 환경 변수를 문서화한다. |
| 테스트 | 상태 전이와 중복 처리, 복원, 엔딩 생성 핵심 테스트를 둔다. |

# 2. 기능 및 비즈니스 명세

## 제약 및 규칙

| 규칙 | 명세 |
|---|---|
| 안전 우선 | `unsafe`, `nuisance`, `illegal`, `embarrassing` 성격의 미션은 후보에서 제외한다. |
| 기존 미션 우선 | 미션은 준비된 콘텐츠에서 먼저 선택한다. 새 미션 자동 생성은 하지 않는다. |
| 환경 태그 | 현재 환경 태그를 모두 만족하는 미션만 기본 후보로 둔다. |
| 차단 태그 | 차단 태그가 하나라도 겹치면 후보에서 제외한다. |
| 완료 중복 방지 | 이미 완료한 미션 ID는 다시 완료 기록에 넣지 않는다. |
| 복선 기록 | 복선 입력이 비어 있으면 저장하지 않는다. |
| 엔딩 조건 | 완료 미션 또는 복선이 1개 이상 있을 때만 엔딩을 생성할 수 있다. |
| 세션 경계 | 하나의 디스코드 서버 또는 지정 채널을 하나의 세션으로 다룬다. |
| 저장 위치 | 세션 상태는 브라우저가 아니라 봇 측 저장소에 저장한다. |

## 기능적 요구사항

| ID | 요구사항 | 입력 | 처리 | 출력 |
|---|---|---|---|---|
| FR-1 | 세션 시작 | 시작 명령, 참가자 정보, 환경 태그 | 새 세션을 만들고 첫 미션을 고른다. | READY 또는 PLAYING 상태 |
| FR-2 | 미션 선택 | 현재 세션 상태와 미션 목록 | 필터링 후 1개 미션을 선택한다. | 현재 미션 또는 범용 미션 |
| FR-3 | 미션 완료 | 현재 미션, 복선 입력, 분위기 | 완료 목록과 복선 목록을 갱신한다. | MISSION_COMPLETE 상태 |
| FR-4 | 체크포인트 이동 | 완료 상태 | 다음 선택 단계를 준비한다. | CHECKPOINT 상태 |
| FR-5 | 범용 미션 | 적합 미션 없음 | 안전한 범용 미션을 제시한다. | EMERGENCY_MISSION 후 PLAYING |
| FR-6 | 엔딩 생성 | 완료 미션과 복선 목록 | 회수 문장을 만든다. | ENDING 상태와 엔딩 텍스트 |
| FR-7 | 세션 저장 | 상태 변경 | 저장소에 직렬화해 저장한다. | 복원 가능한 상태 |
| FR-8 | 세션 초기화 | 리셋 명령 | 저장소와 메모리 상태를 초기화한다. | IDLE 상태 |
| FR-9 | 게임 종료 | 엔딩 완료 후 종료 명령 | 세션을 종료 상태로 바꾼다. | FINISHED 상태 |
| FR-10 | 명령 처리 | 디스코드 interaction | 중복 처리와 상태 검증을 거친다. | 응답 메시지 |

## 상태 전이 규칙

| 현재 | 액션 | 다음 | 실패 조건 | 실패 시 상태 |
|---|---|---|---|---|
| IDLE | START_GAME | READY | 세션 생성 실패 | IDLE |
| READY | SELECT_MISSION | PLAYING | 적합한 미션 없음 | EMERGENCY_MISSION |
| PLAYING | COMPLETE_MISSION | MISSION_COMPLETE | 현재 미션 없음 | PLAYING |
| MISSION_COMPLETE | GO_CHECKPOINT | CHECKPOINT | 완료 저장 실패 | MISSION_COMPLETE |
| CHECKPOINT | SELECT_NEXT_MISSION | PLAYING | 적합한 미션 없음 | EMERGENCY_MISSION |
| EMERGENCY_MISSION | USE_EMERGENCY_MISSION | PLAYING | 범용 미션 없음 | CHECKPOINT |
| CHECKPOINT | GENERATE_ENDING | ENDING | 완료 미션과 복선이 모두 없음 | CHECKPOINT |
| ENDING | FINISH_GAME | FINISHED | 엔딩 텍스트 없음 | ENDING |
| 모든 상태 | RESET_GAME | IDLE | 저장소 초기화 실패 | 기존 상태 유지 |

# 3. 예외 및 운영 정책

## 실패 처리

| 실패 | 처리 |
|---|---|
| 잘못된 명령 | 상태를 바꾸지 않고 현재 가능한 동작만 안내한다. |
| 중복 interaction | 이미 처리된 이벤트로 보고 상태를 바꾸지 않는다. |
| 오래된 버튼/모달 | 최신 세션을 다시 열도록 안내한다. |
| 저장 실패 | 상태 변경을 확정하지 않고 오류를 남긴다. |
| 디스코드 응답 실패 | 1회 재시도 후에도 실패하면 상태는 유지하고 오류 로그를 남긴다. |
| 미션 없음 | 범용 안전 미션으로 전환하거나 체크포인트에 머문다. |
| 엔딩 생성 실패 | 1회 재시도 후 로컬 생성 로직으로 만든다. |

## 재시도 및 멱등성 정책

| 항목 | 정책 |
|---|---|
| interaction 처리 | 동일 interaction ID는 1회만 처리한다. |
| 미션 완료 | 같은 미션 ID 완료 요청은 1번만 반영한다. |
| 복선 기록 | 같은 입력 이벤트는 1번만 저장한다. |
| 디스코드 전송 | 메시지 전송 실패는 1회 재시도한다. |
| 엔딩 생성 | 생성 실패 시 1회 재시도 후 폴백 로직을 사용한다. |
| 리셋 | 여러 번 실행해도 IDLE 상태와 빈 저장소가 유지된다. |

## 로깅 및 모니터링

| 영역 | 명세 |
|---|---|
| 개발 로그 | 상태 전이 실패, 중복 interaction, 저장 실패를 로그로 남긴다. |
| 오류 로그 | 응답 실패, 복원 실패, 엔딩 생성 실패는 오류 로그로 남긴다. |
| 사용자 노출 | 저장 실패, 복원 초기화, 엔딩 폴백은 사용자에게 짧은 상태 메시지로 알린다. |
| 운영 모니터링 | MVP에서는 외부 모니터링을 기본 도입하지 않는다. |

# 4. 완료 및 미결 사항

## 완료 조건

| 기준 | 완료 조건 |
|---|---|
| 시작 | 디스코드 명령으로 세션을 시작할 수 있다. |
| 진행 | 미션 선택, 완료, 체크포인트, 엔딩, 종료가 이어진다. |
| 안전 | 금지 미션이 후보에 들어가지 않는다. |
| 저장 | 재시작 뒤에도 세션이 복원된다. |
| 중복 방지 | 중복 interaction으로 완료/복선이 두 번 저장되지 않는다. |
| 폴백 | 적합 미션이 없을 때 범용 미션 경로가 동작한다. |
| 빌드 | 개발 실행과 프로덕션 실행이 가능하다. |
| 검증 | 핵심 상태 전이 테스트가 통과한다. |
