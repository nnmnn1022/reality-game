# 전체 요약

## 요약

- `reality-game`은 기존 웹앱을 버리고 디스코드에서 진행하는 게임으로 다시 만든다.
- 플레이어는 디스코드 안에서 미션을 받고, 선택을 남기고, 그 기록이 마지막 이야기로 회수된다.
- 이번 작업은 화면 꾸미기보다 게임 흐름과 디스코드 상호작용을 먼저 만드는 데 초점을 둔다.

## 상세

| 구분 | 내용 |
|---|---|
| 제품 성격 | 디스코드 안에서 함께 진행하는 이야기형 게임이다. |
| 핵심 경험 | 처음에는 별 의미 없어 보이는 행동이 마지막에 하나의 이야기로 연결된다. |
| 서비스 형태 | 웹페이지가 아니라 디스코드 세션에서 플레이한다. |
| 운영 방식 | 게임 상태와 복선은 저장해 두었다가 다시 이어서 볼 수 있어야 한다. |

# 문제

## 요약

- 기존 웹앱은 디스코드 게임의 진행 방식과 맞지 않는다.
- 단순 정보 제공이 아니라, 플레이어가 직접 선택하고 기록하는 구조가 필요하다.
- 안전하지 않거나 민폐가 되는 미션은 처음부터 제외해야 한다.

## 상세

| 문제 | 설명 |
|---|---|
| 진행 방식 | 웹앱처럼 한 화면에 정보를 보여주는 방식으로는 디스코드 게임을 자연스럽게 운영하기 어렵다. |
| 참여성 | 모든 참가자가 대화와 선택에 참여할 수 있어야 한다. |
| 이야기성 | 남긴 기록이 마지막에 이야기로 돌아와야 한다. |
| 안전성 | 위험, 불법, 민폐, 창피함 강요를 배제해야 한다. |

# 원인

## 요약

- 제품 방향이 웹앱에서 디스코드 게임으로 바뀌었다.
- 따라서 기존 웹앱 구현을 그대로 쓰는 방식은 맞지 않는다.

## 상세

| 항목 | 내용 |
|---|---|
| 방향 변화 | 기존 웹앱을 새 제품의 중심으로 두지 않는다. |
| 현재 한계 | 브라우저 입력과 로컬 저장에 기대는 구조는 디스코드 게임 경험을 설명하지 못한다. |
| 해결 방향 | 디스코드 상호작용을 중심으로 게임 흐름을 다시 잡는다. |

# 현재(AS-IS)

## 요약

- 현재는 웹앱 형태의 게임 화면과 브라우저 저장 구조가 있다.
- 디스코드에서 바로 플레이할 수 있는 게임 흐름은 없다.
- 운영과 복원도 브라우저 기준으로 되어 있다.

## 상세

| 영역 | 현재 상태 |
|---|---|
| 실행 | 웹페이지를 여는 방식이다. |
| 입력 | 참가자 정보와 태그, 진행 기록을 폼으로 받는다. |
| 저장 | 브라우저에 상태를 저장한다. |
| 플레이 흐름 | 화면 중심으로 시작, 진행, 엔딩을 처리한다. |
| 배포 | 디스코드 게임 운영 방식은 준비되어 있지 않다. |

# 목표(TO-BE)

## 요약

- 디스코드에서 게임을 시작하고 끝까지 진행할 수 있어야 한다.
- 미션은 안전 기준과 현재 상황에 맞게 선택되어야 한다.
- 완료 기록과 복선이 누적되어 엔딩에 반영되어야 한다.
- 재시작 후에도 세션을 이어갈 수 있어야 한다.

## 상세

| 목표 | 성공 상태 |
|---|---|
| 게임 시작 | 디스코드 명령으로 세션을 시작한다. |
| 미션 진행 | 현재 미션과 다음 행동을 디스코드에서 확인하고 진행한다. |
| 복선 기록 | 선택, 감정, 관찰, 물건 같은 기록을 남긴다. |
| 엔딩 | 누적된 기록이 이야기형 엔딩으로 이어진다. |
| 복원 | 저장된 세션을 다시 불러올 수 있다. |
