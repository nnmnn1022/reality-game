# TO-BE

## 요약

- `reality-game`은 더 이상 웹앱이 아니라 디스코드 안에서 진행되는 그룹 게임으로 제공한다.
- 플레이어는 디스코드 채널에서 게임을 시작하고, 미션을 받고, 결과를 남기고, 복선을 축적해 엔딩을 본다.
- 시스템은 이야기 자체를 고정해서 보여주기보다, 진행 단계와 경험 균형을 관리하는 엔진으로 동작한다.
- 기존 웹 화면은 제거 대상으로 보고, 디스코드 상호작용만을 기본 경험으로 삼는다.

## 상세

| 영역 | TO-BE |
|---|---|
| 진입점 | 디스코드 명령으로 게임 세션을 시작한다. |
| 진행 단위 | 서버 또는 지정 채널 단위의 하나의 게임 세션을 운영한다. |
| 입력 방식 | 명령, 버튼, 선택지, 모달 입력으로 플레이어 행동을 받는다. |
| 상태 흐름 | 준비, 진행, 체크포인트, 완료, 엔딩, 종료의 흐름을 명시적으로 유지한다. |
| 미션 | 환경 태그와 세션 상태를 기준으로 안전한 미션만 노출한다. |
| 복선 | 미션 결과, 선택, 감정, 관찰을 구조화해 축적한다. |
| 엔딩 | 축적된 복선을 회수해 한 편의 이야기로 마무리한다. |
| 운영 | 재시작 후에도 세션을 복원할 수 있는 저장 방식을 사용한다. |

## 상태 전이

| 현재 상태 | 트리거 | 다음 상태 | 조건 |
|---|---|---|---|
| IDLE | `/start` | READY | 세션 생성 성공 |
| READY | `미션 시작` | PLAYING | 시작 가능한 미션이 선택됨 |
| PLAYING | `미션 완료` | MISSION_COMPLETE | 현재 미션 존재 |
| MISSION_COMPLETE | `체크포인트 이동` | CHECKPOINT | 완료 결과 저장 성공 |
| CHECKPOINT | `다음 미션 선택` | PLAYING | 적합한 미션 선택 성공 |
| CHECKPOINT | `범용 미션 사용` | PLAYING | 범용 미션 존재 |
| CHECKPOINT | `엔딩 생성` | ENDING | 완료 미션 또는 복선이 1개 이상 |
| ENDING | `게임 종료` | FINISHED | 엔딩 텍스트 생성 성공 |
| 모든 상태 | `리셋` | IDLE | 세션 폐기 성공 |

## 트랜잭션 경계

| 작업 | 경계 |
|---|---|
| 세션 시작 | 세션 생성, 기본 참가자/환경 태그 설정, 최초 상태 기록을 하나의 갱신으로 처리한다. |
| 미션 배정 | 후보 계산과 미션 확정을 하나의 처리 단위로 묶는다. |
| 미션 완료 | 현재 미션 종료, 복선 기록, 상태 전이를 한 번에 저장한다. |
| 엔딩 생성 | 누적 복선 스냅샷을 기준으로 엔딩을 생성하고 상태에 반영한다. |

# 설계 결정

## 요약

- 디스코드 상호작용은 명령과 버튼 중심으로 설계한다.
- 세션 상태는 디스코드 실행 환경과 분리된 영속 저장소에 보관한다.
- 기존 웹앱은 새 구현의 기준이 아니라 폐기 대상이며, 새 서비스는 봇 중심으로 시작한다.
- 명령 중복, 메시지 지연, stale interaction 같은 디스코드 특유의 실패를 전제로 처리한다.

## 상세

| 결정 | 내용 | 이유 |
|---|---|---|
| 진입 방식 | 게임 시작은 명령 기반으로 연다. | 대화 채널 안에서 세션을 명확히 시작해야 하기 때문이다. |
| 상호작용 | 미션 선택과 진행은 버튼/선택지, 텍스트 입력은 모달로 받는다. | 채팅 오염을 줄이고, 조작을 단순하게 유지한다. |
| 세션 단위 | 하나의 서버 또는 지정 채널을 하나의 세션 경계로 본다. | 그룹 게임의 문맥을 분리하기 쉽다. |
| 저장 방식 | 세션과 복선은 영속 저장소에 직렬화한다. | 프로세스 재시작 후 복원이 가능해야 한다. |
| 기존 웹앱 | 웹 UI와 브라우저 localStorage 전제는 제거한다. | 새 제품이 디스코드 게임이기 때문이다. |
| 미션 선택 | 환경 태그, 완료 이력, 안전 태그를 기준으로 후보를 계산한다. | 기존 제품의 안전 원칙을 유지해야 한다. |
| 엔딩 생성 | 복선과 완료 미션을 기반으로 로컬 생성 로직을 기본값으로 둔다. | 외부 AI 의존 없이도 게임이 끝나야 한다. |
| 상태 갱신 | 모든 주요 액션은 상태 저장이 성공한 뒤에 사용자에게 확정 응답한다. | 디스코드에서 중복 클릭과 지연이 흔하기 때문이다. |

## 실패 처리

| 상황 | 처리 |
|---|---|
| 중복 명령 또는 중복 클릭 | 이미 처리된 이벤트로 판단되면 상태를 바꾸지 않고 짧은 확인 메시지만 보낸다. |
| 오래된 버튼/모달 | 현재 상태와 맞지 않으면 거부하고 최신 세션 안내를 보낸다. |
| 미션 없음 | 범용 안전 미션으로 전환하거나 세션을 체크포인트에 둔다. |
| 저장 실패 | 상태 변경을 확정하지 않고 오류 응답을 남긴다. |
| 엔딩 생성 실패 | 1회 재시도 후 로컬 폴백 엔딩으로 전환한다. |
| 디스코드 API 오류 | 재전송이 가능한 경우 1회 재시도하고, 그 외에는 세션 상태를 보존한 채 사용자에게 실패를 알린다. |

## 재시도 및 멱등성

| 항목 | 정책 |
|---|---|
| 명령 처리 | 동일 interaction ID는 1회만 처리한다. |
| 미션 완료 | 같은 미션 완료 요청이 반복돼도 완료 목록에는 1회만 반영한다. |
| 복선 기록 | 동일 입력 제출은 고유 이벤트 ID를 붙여 중복 저장을 방지한다. |
| 메시지 전송 | 일시적 전송 실패는 1회 재시도한다. |
| 엔딩 생성 | 엔딩 생성 중간 실패 시 이전 상태를 유지하고, 성공한 경우에만 상태를 ENDING으로 바꾼다. |

# 리스크

## 요약

- 디스코드 UX 문서가 아직 없어 명령/버튼/모달 조합은 임시 가정이다.
- 채널 단위 세션은 그룹 플레이에 적합하지만, 멀티 세션 운영 정책이 추가로 필요할 수 있다.
- 외부 저장소를 붙이지 않으면 재시작 복원과 운영 안정성이 제한된다.

## 상세

| 리스크 | 영향 | 대응 |
|---|---|---|
| UX 미정 | 실제 사용자 흐름과 명령 구조가 바뀔 수 있다. | 현재 설계는 명령/버튼/모달을 기본 가정으로 두고, 후속 UX 문서가 오면 조정한다. |
| 세션 경계 충돌 | 같은 채널에서 여러 게임을 동시에 운영하기 어렵다. | 기본은 채널 1세션으로 두고, 다중 세션은 후속 범위로 둔다. |
| 저장소 단순성 | 영속 저장소가 약하면 재시작 복원이 불안정하다. | 프로세스 메모리만 쓰지 않고 직렬화 가능한 저장 계층을 둔다. |
| 기존 웹앱 폐기 | 코드 제거 범위가 넓어질 수 있다. | 실행 준비가 되면 레거시 UI와 상태 코드를 정리하고 새 봇 구조로 교체한다. |
