# 분석

## 요약

- 현재 구현은 Next.js 기반 웹앱으로 동작한다.
- 게임은 브라우저 화면에서 시작하고, 상태는 클라이언트 localStorage에 저장된다.
- 디스코드 전용 흐름, 메시지 상호작용, 봇 이벤트 구조는 아직 구현되어 있지 않다.
- 따라서 전환 영향은 화면 구조 전체, 입력 방식, 상태 전달 경로, 저장 방식, 운영 설정까지 넓다.

## 상세

| 영역 | 현재 상태 | 근거 |
|---|---|---|
| 실행 방식 | `next dev`, `next build`, `next start`를 쓰는 웹앱이다. | `package.json` |
| 첫 화면 | 단일 `GameConsole` 컴포넌트가 루트 화면을 차지한다. | `src/app/page.tsx` |
| 사용자 입력 | 참가자명, 환경 태그, 복선 문장, 분위기를 폼 입력으로 받는다. | `src/components/GameConsole.tsx` |
| 진행 방식 | 시작, 초기화, 미션 완료, 체크포인트, 범용 미션, 엔딩 생성, 종료 버튼으로 상태를 바꾼다. | `src/components/GameConsole.tsx`, `src/lib/game.ts` |
| 상태 저장 | 게임 상태를 브라우저 `localStorage`에 직렬화해 보존한다. | `src/components/GameConsole.tsx`, `src/lib/game.ts` |

| 화면 성격 | 현재 UI는 웹 폼과 패널 중심이며, 디스코드 메시지 UX와 직접 연결되는 구조가 아니다. | `src/components/GameConsole.tsx`, `src/app/globals.css` |

## AS-IS

## 요약

- 현재 제품은 현실 미션 게임 웹서비스 MVP다.
- 메인 화면에서 게임을 시작하고, 상태를 진행하며, 엔딩을 생성하는 구조다.
- 미션 선택은 환경 태그와 완료 이력을 기준으로 수행된다.
- 저장은 브라우저 내부에서만 이뤄진다.
- 디스코드 게임 전환에 필요한 봇, 채널, 메시지 흐름은 아직 없다.

## 상세

| 항목 | 현재 동작 |
|---|---|
| 시작 상태 | `SETUP`에서 참가자와 태그를 받아 `PLAYING`으로 전환한다. |
| 미션 선택 | 태그 조건과 완료 여부를 기준으로 미션 풀에서 선택한다. |
| 미션 완료 | 현재 미션을 완료하고, 입력된 복선 문장을 기록한다. |
| 체크포인트 | 완료 후 다음 미션으로 넘기거나 범용 미션으로 전환한다. |
| 엔딩 | 완료 미션과 복선이 있으면 엔딩 문장을 만든다. |
| 종료 | 엔딩 상태에서만 게임 종료 상태로 바꾼다. |
| 복원 | 새로고침 뒤에도 `localStorage` 값으로 상태를 복원한다. |
| UI 구조 | 하나의 웹 페이지에 시작, 현재 미션, 진행, 엔딩, 상태, 미션 목록, 복선 영역이 함께 노출된다. |

## 영향 범위

| 범위 | 영향 |
|---|---|
| 제품 정의 | 웹서비스가 아니라 디스코드 게임이라는 전제로 다시 잡아야 한다. 기존 웹앱은 재사용 전제가 아니다. |
| 인터랙션 | 폼 입력 중심 UI를 디스코드 메시지/명령/버튼 중심 흐름으로 바꿔야 한다. |
| 상태 관리 | 브라우저 localStorage 기반 저장은 디스코드 실행 환경에 맞게 재검토가 필요하다. |
| 콘텐츠 전달 | 현재의 단일 페이지 정보 배치는 디스코드 대화 흐름에 맞지 않는다. |
| 운영 설정 | 웹 배포 설정만으로는 디스코드 봇 운영을 설명할 수 없다. |
