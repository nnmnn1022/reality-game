# TO-BE

## 요약

- 현실 미션 게임 MVP는 브라우저에서 실행되는 웹서비스로 제공한다.
- JSON 기반 미션 데이터와 클라이언트 상태를 사용해 설정, 진행, 체크포인트, 미션 완료, 엔딩까지 이어지는 플레이 흐름을 제공한다.
- AI 연동은 서비스 가능한 경계만 마련하고, MVP에서는 로컬 규칙 기반 엔딩 생성과 모의 AI 응답을 기본값으로 둔다.
- 개발 실행, 프로덕션 빌드, 환경 변수 예시, 배포 플랫폼 설정을 포함해 실제 서비스 준비가 가능한 형태로 구성한다.

## 상세

| 영역 | TO-BE |
|---|---|
| 사용자 경험 | 첫 화면에서 참가자와 환경 태그를 설정하고, 이후 현재 미션, 체크포인트, 복선, 엔딩을 하나의 플레이 콘솔에서 진행한다. |
| 콘텐츠 | 미션은 JSON 데이터로 관리하며, 안전/민폐 방지/환경 태그/복선 태그를 기준으로 선택한다. |
| 상태 | 게임 상태는 명시적 상태 머신으로 전환하며, 사용자의 진행 버튼과 복선 입력에 따라 갱신한다. |
| 복선 | 미션 완료 시 선택, 감정, 관찰, 물건 같은 복선 기록을 누적하고 엔딩 생성에 사용한다. |
| 엔딩 | 수집된 복선을 조합해 이야기형 엔딩을 생성한다. 실제 AI 키가 없을 때도 서비스가 동작해야 한다. |
| 서비스 | Next.js 기반 웹 앱으로 개발 서버와 프로덕션 빌드를 지원하고, 배포 플랫폼에서 실행 가능한 설정을 제공한다. |

## 상태 전이

| 현재 상태 | 트리거 | 다음 상태 | 조건 |
|---|---|---|---|
| SETUP | 게임 시작 | PLAYING | 참가자와 환경 태그가 기본값 이상으로 존재한다. |
| PLAYING | 체크포인트 도달 | CHECKPOINT | 현재 미션이 존재하고 진행 시간이 체크포인트 기준에 도달했거나 사용자가 이동한다. |
| CHECKPOINT | 미션 선택 | PLAYING | 환경 태그와 상태에 맞는 미션이 선택된다. |
| PLAYING | 미션 완료 | MISSION_COMPLETE | 복선 기록 입력 여부와 무관하게 완료 처리가 가능하되, 복선 입력을 권장한다. |
| MISSION_COMPLETE | 다음 미션 | CHECKPOINT | 완료 미션이 기록되고 다음 선택 단계로 이동한다. |
| CHECKPOINT | 적합 미션 없음 | EMERGENCY_MISSION | 필터 결과가 비었거나 안전 조건을 만족하는 미션이 부족하다. |
| EMERGENCY_MISSION | 범용 미션 선택 | PLAYING | 준비된 범용 안전 미션을 선택한다. |
| CHECKPOINT | 엔딩 생성 | ENDING | 최소 1개 이상의 완료 미션 또는 복선 기록이 존재한다. |
| ENDING | 종료 | FINISHED | 엔딩 문장이 생성되어 사용자에게 표시된다. |

## 트랜잭션 경계

| 작업 | 경계 |
|---|---|
| 게임 시작 | 참가자, 환경 태그, 초기 상태, 첫 미션 선택을 하나의 상태 갱신으로 처리한다. |
| 미션 완료 | 완료 미션 ID, 피로도/분위기 변화, 복선 기록을 하나의 상태 갱신으로 처리한다. |
| 다음 미션 선택 | 현재 상태와 완료 목록을 기준으로 후보를 계산하고, 선택된 미션만 상태에 반영한다. |
| 엔딩 생성 | 현재까지의 복선 스냅샷을 기준으로 엔딩을 생성하며, 생성 결과를 종료 전 상태에 보관한다. |

# 설계 결정

## 요약

- 프론트엔드는 기획서 기준에 맞춰 Next.js를 사용한다.
- MVP 저장소는 JSON과 브라우저 로컬 상태를 사용하고, 서버 데이터베이스는 도입하지 않는다.
- AI API 키가 없어도 동작하는 로컬 엔딩 생성기를 기본으로 두고, 실제 AI 연동은 환경 변수로 켤 수 있는 경계만 둔다.
- 배포 준비는 정적/서버리스 호스팅이 쉬운 구조를 우선한다.

## 상세

| 결정 | 내용 | 이유 |
|---|---|---|
| 앱 구조 | Next.js App Router 기반 단일 웹 앱으로 구성한다. | 기획서의 React/Next.js 방향을 따르고, 서비스 배포가 쉽다. |
| 데이터 저장 | `data` 영역의 JSON 미션 데이터와 클라이언트 상태를 사용한다. | MVP에서 데이터베이스를 제외한다는 범위와 맞다. |
| 상태 관리 | 별도 전역 상태 라이브러리 없이 명시적 reducer/상태 전이 함수로 관리한다. | 상태 전이가 핵심이므로 테스트 가능한 순수 함수 경계를 둔다. |
| 미션 선택 | 환경 태그, 차단 태그, 완료 목록, 단계, 안전 플래그를 기준으로 후보를 계산한다. | 안전과 환경 적합성을 우선한다. |
| AI 경계 | 엔딩 생성 인터페이스를 분리하고 기본 구현은 로컬 규칙 기반으로 둔다. | API 키 없이도 서비스 가능해야 하며, 실제 AI 연동은 후속 확장 가능해야 한다. |
| 서비스 설정 | `.env.example`, README 실행 안내, 프로덕션 빌드 스크립트, 배포 설정 파일을 포함한다. | “실제로 서비스할 수 있도록 설정”한다는 사용자 범위를 만족한다. |
| 검증 | 타입 검사, 린트, 빌드, 핵심 상태 전이 테스트를 검증 기준으로 둔다. | 앱 동작과 서비스 준비 상태를 함께 확인한다. |

## 오류 처리

| 상황 | 처리 |
|---|---|
| 적합한 미션이 없음 | EMERGENCY_MISSION으로 전환하고 범용 안전 미션을 제시한다. |
| 복선 입력이 비어 있음 | 미션 완료는 허용하되, 엔딩에서는 완료 미션과 선택 기록을 중심으로 생성한다. |
| 환경 태그가 부족함 | 기본 태그를 사용해 시작 가능하게 하고, 화면에서 태그 추가를 유도한다. |
| AI API 키가 없음 | 로컬 엔딩 생성기를 사용한다. |
| AI 응답 실패 | 재시도 1회 후 로컬 엔딩 생성기로 폴백한다. |
| 잘못된 상태 전이 | 상태를 변경하지 않고 사용자에게 현재 가능한 액션만 노출한다. |

## 재시도, 롤백, 멱등성

| 항목 | 설계 |
|---|---|
| 재시도 | 외부 AI 호출은 1회 재시도하고 실패하면 로컬 생성기로 폴백한다. 미션 선택은 같은 상태에서 반복 실행해도 같은 후보군을 유지한다. |
| 롤백 | 미션 완료 처리 중 오류가 발생하면 완료 미션과 복선 기록을 모두 반영하지 않고 이전 상태를 유지한다. |
| 멱등성 | 같은 미션 완료 액션이 반복되어도 완료 미션 ID는 중복 저장하지 않는다. 같은 복선 제출은 생성 시각 또는 고유 ID로 구분한다. |

# 리스크

## 요약

- 실제 AI 연동은 API 키와 비용, 응답 안정성에 따라 후속 조정이 필요하다.
- JSON 기반 MVP는 다중 사용자 동시 플레이 저장에는 한계가 있다.
- 미션 콘텐츠 수가 적으면 추천 품질과 엔딩 품질이 제한될 수 있다.

## 상세

| 리스크 | 영향 | 대응 |
|---|---|---|
| 실제 AI 호출 불확실성 | 배포 환경에서 API 키가 없거나 호출 실패 시 엔딩 생성이 중단될 수 있다. | 로컬 엔딩 생성기를 기본 폴백으로 둔다. |
| 콘텐츠 부족 | 반복 플레이에서 미션 다양성이 낮아진다. | 샘플 미션은 MVP 검증 가능한 최소 수량으로 만들고, 데이터 구조는 확장 가능하게 둔다. |
| 브라우저 상태 저장 한계 | 새로고침이나 기기 변경 시 상태 유지가 제한될 수 있다. | MVP에서는 localStorage 저장을 사용하고, 데이터베이스 저장은 후속 범위로 둔다. |
| 배포 플랫폼 차이 | 플랫폼마다 환경 변수와 빌드 설정 방식이 다를 수 있다. | 표준 Node/Next.js 빌드를 기준으로 하고, Vercel 호환 설정을 제공한다. |
| 안전 규칙 누락 | 부적절한 미션이 사용자에게 노출될 수 있다. | 미션 데이터에 차단 태그와 안전 카테고리를 두고 선택 함수에서 제외한다. |
