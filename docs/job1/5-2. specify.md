# 1. 시스템 영향 및 범위

## 배경 및 목표

| 항목 | 명세 |
|---|---|
| 배경 | `/home/umoo/reality-game`에는 현실 미션 게임 기획 문서가 있고, 실행 가능한 앱은 아직 없다. |
| 목표 | Next.js 기반 웹서비스 MVP를 만들고, 사용자가 설정부터 미션 진행, 복선 기록, 엔딩 생성까지 브라우저에서 수행하게 한다. |
| 서비스 목표 | 개발 서버 실행, 프로덕션 빌드, 환경 변수 예시, 배포 설정을 포함한다. |
| AI 목표 | 실제 AI API 키가 없어도 로컬 엔딩 생성기로 동작한다. |

## 범위

| 포함 | 제외 |
|---|---|
| Next.js 웹 앱 생성 | 관리자 미션 편집기 |
| JSON 기반 미션 데이터 | SQLite/PostgreSQL 저장 |
| 브라우저 localStorage 기반 게임 상태 저장 | 사용자 계정/로그인 |
| 미션 선택, 완료, 복선 기록, 엔딩 생성 | 고도화 추천 시스템 |
| 로컬 엔딩 생성기와 AI 연동 경계 | 실제 결제/위치 기반 운영 기능 |
| 개발/빌드/배포 준비 설정 | 사용자 제작 미션 |

## 시스템 영향

| 영역 | 영향 |
|---|---|
| 프로젝트 구조 | Next.js 앱, 데이터, 상태 전이 로직, 미션 저장소, 엔딩 생성 경계가 추가된다. |
| 사용자 화면 | 플레이 콘솔, 설정 영역, 미션 카드, 복선 입력, 엔딩 영역이 추가된다. |
| 데이터 | 샘플 미션 JSON과 초기 게임 상태 구조가 추가된다. |
| 저장 | 브라우저 localStorage에 게임 상태를 저장하고 복원한다. |
| 운영 | `.env.example`, README, 배포 설정 파일을 통해 서비스 준비 정보를 제공한다. |

# 2. 기능 및 비즈니스 명세

## 제약 및 규칙

| 규칙 | 명세 |
|---|---|
| 안전 우선 | `unsafe`, `nuisance`, `illegal`, `embarrassing` 성격을 가진 미션은 선택 후보에서 제외한다. |
| 기존 미션 우선 | 미션은 `data/missions.json`에서 먼저 선택한다. 새 미션 자동 생성은 MVP에서 하지 않는다. |
| 환경 태그 | 미션의 `requiredTags`가 모두 현재 환경 태그에 포함되어야 기본 후보가 된다. |
| 차단 태그 | 미션의 `blockedTags` 중 하나라도 현재 환경 태그에 포함되면 후보에서 제외한다. |
| 완료 중복 방지 | 이미 완료한 미션 ID는 다시 선택하지 않는다. |
| 복선 기록 | 미션 완료 시 복선 입력이 없더라도 완료 가능하다. 빈 복선은 저장하지 않는다. |
| 엔딩 조건 | 완료 미션이 1개 이상이거나 복선이 1개 이상 있으면 엔딩 생성 버튼을 활성화한다. |
| 저장 위치 | MVP 상태는 localStorage에 저장한다. 서버 저장은 하지 않는다. |
| AI 키 부재 | AI 환경 변수가 없으면 로컬 엔딩 생성기를 사용한다. |

## 기능적 요구사항

| ID | 요구사항 | 입력 | 처리 | 출력 |
|---|---|---|---|---|
| FR-1 | 게임 시작 | 참가자 이름 목록, 환경 태그 목록 | 초기 상태를 만들고 첫 미션을 선택한다. | PLAYING 상태와 현재 미션 |
| FR-2 | 미션 선택 | 현재 상태, 미션 목록 | 필터링 후 후보를 점수화해 1개를 선택한다. | 선택된 미션 또는 EMERGENCY_MISSION |
| FR-3 | 미션 완료 | 현재 미션, 복선 입력, 감정 | 완료 목록과 복선 목록을 한 번에 갱신한다. | MISSION_COMPLETE 상태 |
| FR-4 | 다음 체크포인트 | 완료 상태 | CHECKPOINT로 이동하고 다음 미션 선택을 준비한다. | CHECKPOINT 상태 |
| FR-5 | 범용 미션 | 적합 미션 없음 | 안전한 범용 미션을 선택한다. | EMERGENCY_MISSION 후 PLAYING |
| FR-6 | 엔딩 생성 | 완료 미션과 복선 목록 | 로컬 생성기 또는 AI 경계를 통해 엔딩 문장을 만든다. | ENDING 상태와 엔딩 텍스트 |
| FR-7 | 상태 저장 | 상태 변경 | localStorage에 직렬화해 저장한다. | 새로고침 후 복원 가능한 상태 |
| FR-8 | 상태 초기화 | 리셋 액션 | localStorage와 메모리 상태를 초기값으로 되돌린다. | SETUP 상태 |
| FR-9 | 서비스 빌드 | 소스 코드 | 프로덕션 빌드를 수행한다. | 빌드 성공 |

## 상태 전이 규칙

| 현재 | 액션 | 다음 | 실패 조건 | 실패 시 상태 |
|---|---|---|---|---|
| SETUP | START_GAME | PLAYING | 참가자가 0명이고 기본 참가자 생성도 실패 | SETUP |
| PLAYING | COMPLETE_MISSION | MISSION_COMPLETE | 현재 미션이 없음 | PLAYING |
| MISSION_COMPLETE | GO_CHECKPOINT | CHECKPOINT | 없음 | 해당 없음 |
| CHECKPOINT | SELECT_NEXT_MISSION | PLAYING | 적합 미션 없음 | EMERGENCY_MISSION |
| EMERGENCY_MISSION | USE_EMERGENCY_MISSION | PLAYING | 범용 미션 데이터 없음 | CHECKPOINT |
| CHECKPOINT | GENERATE_ENDING | ENDING | 완료 미션과 복선이 모두 없음 | CHECKPOINT |
| ENDING | FINISH_GAME | FINISHED | 엔딩 텍스트 없음 | ENDING |
| 모든 상태 | RESET_GAME | SETUP | 없음 | 해당 없음 |

# 3. 예외 및 운영 정책

## 실패 처리

| 실패 | 처리 |
|---|---|
| 미션 JSON 파싱 실패 | 앱 빌드 또는 테스트에서 실패로 처리한다. 런타임에서는 빈 배열로 숨기지 않는다. |
| 적합 미션 없음 | `EMERGENCY_MISSION` 상태로 전환하고 범용 안전 미션을 표시한다. |
| 범용 미션 없음 | 사용자가 환경 태그를 수정하도록 안내하고 CHECKPOINT 상태를 유지한다. |
| localStorage 저장 실패 | 화면 상태는 유지하고, 사용자에게 브라우저 저장 실패 메시지를 표시한다. |
| localStorage 복원 실패 | 손상된 저장값을 삭제하고 SETUP 상태로 시작한다. |
| AI 호출 실패 | 1회 재시도 후 로컬 엔딩 생성기로 폴백한다. |
| 잘못된 상태 액션 | 상태를 변경하지 않고 현재 상태에서 가능한 액션만 UI에 노출한다. |

## 재시도 및 멱등성 정책

| 항목 | 정책 |
|---|---|
| AI 호출 | 타임아웃 또는 오류 발생 시 1회 재시도한다. 2회 실패하면 로컬 엔딩 생성기를 사용한다. |
| 미션 완료 | 같은 미션 ID 완료 요청이 반복되면 완료 목록에는 1번만 저장한다. |
| 복선 저장 | 동일 액션에서 생성된 복선은 고유 ID를 부여해 저장한다. 빈 문자열은 저장하지 않는다. |
| 상태 저장 | 상태 변경 후 한 번 저장한다. 저장 실패가 발생해도 메모리 상태를 되돌리지 않는다. |
| 롤백 | 미션 완료 처리 중 예외가 발생하면 완료 미션과 복선 기록 모두 이전 상태를 유지한다. |
| 리셋 | 리셋은 멱등적이다. 여러 번 실행해도 SETUP 상태와 빈 저장소가 유지된다. |

## 로깅 및 모니터링

| 영역 | 명세 |
|---|---|
| 개발 로그 | 상태 전이 실패와 저장 실패는 `console.warn`으로 남긴다. |
| 오류 로그 | 엔딩 생성 실패와 저장소 복원 실패는 `console.error`로 남긴다. |
| 운영 모니터링 | MVP에서는 외부 모니터링 서비스를 붙이지 않는다. README에 추후 Vercel Analytics 또는 Sentry 연결 지점을 기록한다. |
| 사용자 노출 | 저장 실패, 복원 초기화, 엔딩 폴백은 화면의 상태 메시지로 표시한다. |
